# 百年前新聞オンライン 現状監査レポート

作成日: 2026-02-17

## 0. 対象と前提
- 対象: `index.html` / `script.js` / `style.css` / `sw.js` / `manifest.json`
- 画面は「記事一覧 + タグ絞り込み + サムネイル + PWA化」を備えた単一ページ構成。

---

## 1. 複数観点のチェックと得点化（100点満点）

### 1-1. UX・情報設計: **74 / 100**
**良い点**
- ジャンル・タグ・キーワード・並び替えがあり、探索手段が多い。
- 「記事 / タグ一覧 / サムネイル一覧」タブがあり利用文脈を切替できる。
- 無限スクロールや「上に戻る」ボタンなど閲覧補助がある。

**懸念**
- 右カラムの「サムネイル一覧」と左カラムのタブ「サムネイル一覧」が重複し、導線が冗長。
- `scrollSentinel` 参照箇所が複数あるのに HTML 上で要素定義が見当たらず、0件時や初期描画時に不整合リスク。
- 取得失敗時に `#app` へメッセージ出力しているが、実DOMに `id="app"` が存在せずエラー表示が期待通りに機能しない可能性。

### 1-2. パフォーマンス: **67 / 100**
**良い点**
- 初期表示後の分割描画（`BATCH_SIZE`）があり、全件一括よりは軽い。
- Service Workerで静的資産をキャッシュして再訪問を高速化。

**懸念**
- クライアント側で全記事を一括取得してから絞り込み/並び替えしており、データ増加時にCPU・メモリ負荷が上がる。
- 文字列連結で `innerHTML +=` を多用しており、DOM再構築コスト・XSS耐性の面で不利。
- `CACHE_VERSION = Date.now()` は毎回キャッシュ名が変わるため、更新通知は確実だがキャッシュ効率は低下しやすい。

### 1-3. 保守性・コード品質: **62 / 100**
**良い点**
- 機能ごとの関数分割は一定程度されている。
- 変数名・コメントが用途を示している箇所が多い。

**懸念**
- 単一 `script.js` に状態・描画・イベント・通信が集中し、責務分離が不十分。
- グローバル状態 (`window.filteredArticles` 等) 依存があり、副作用追跡が難しい。
- 例外時のフォールバックが不足（null要素参照の可能性）。

### 1-4. 信頼性・エラーハンドリング: **55 / 100**
**良い点**
- API応答 `ok` 判定は実装済み。
- Service Worker登録失敗時にログ出力あり。

**懸念**
- 上述の `#app` 不在により失敗時UXが壊れる可能性。
- `document.getElementById("scrollSentinel")` を存在確認なしで操作しており、ランタイムエラー余地がある。
- ネットワーク遅延・タイムアウト・再試行方針が未定義。

### 1-5. アクセシビリティ: **58 / 100**
**良い点**
- ボタン主体UIで基本操作は可能。
- 画像は `loading="lazy"` 指定あり。

**懸念**
- タブUIに `aria-selected` / `role="tablist"` 等のセマンティクスが不足。
- モーダルのフォーカストラップ・Esc閉じる・スクリーンリーダー配慮が不明。
- 色コントラストやキーボード操作性を保証する記述が見えない。

### 1-6. PWA・配信設計: **71 / 100**
**良い点**
- `manifest.json` と Service Worker があり、ホーム画面追加導線もある。
- iOS向け案内表示を実装。

**懸念**
- フェッチ戦略が簡素で、APIレスポンスのキャッシュ戦略/失効戦略が未整理。
- 更新通知はあるが、SW更新フロー（skipWaiting/clientsClaim連携等）が最適化余地あり。

### 1-7. セキュリティ: **60 / 100**
**良い点**
- HTTPSエンドポイントを使用。

**懸念**
- 記事本文をHTML文字列で挿入する実装は、データ源によりXSSリスク。
- CSP等の防御ヘッダ・入力サニタイズ方針が不明。

## 2. 現状の問題点（総合）

### 2-1. 重要度: 高（先に対応）
1. **エラー表示先ID不整合**: 失敗時メッセージ出力先 `#app` が存在しない。
2. **Sentinel参照の堅牢性不足**: `scrollSentinel` 不在時の保護がない。
3. **描画方式の安全性**: `innerHTML` 連結中心でXSS・性能の両面リスク。
4. **単一巨大スクリプト**: 機能拡張時の不具合混入確率が高い。

### 2-2. 重要度: 中
1. **UI導線の重複/混乱**（サムネイル導線の二重化）。
2. **アクセシビリティ未整備**（タブ/モーダル/キーボード操作）。
3. **キャッシュ戦略の粗さ**（毎回新キャッシュ名）。
4. **監視・計測不足**（エラー率、読み込み時間、検索利用率などKPIがない）。

### 2-3. 重要度: 低
1. **設計ドキュメント不足**（状態遷移やデータモデル定義）。
2. **テスト自動化不足**（最低限の回帰確認が手動依存）。

## 3. 今後の開発方針（実行手順）

### Phase 0（1〜2日）: 緊急安定化
- [ ] `#app` 不整合を修正（適切なエラー表示領域を追加/置換）。
- [ ] `scrollSentinel` 参照前ガードを追加（存在しない場合でも落ちない）。
- [ ] 主要箇所で null チェックを徹底。
- [ ] 最低限の静的チェックをCI化（`node --check script.js` など）。

**完了条件**
- API失敗時でも画面がクラッシュせず、ユーザーに明確な再試行導線が表示される。

### Phase 1（1週間）: UX/アクセシビリティ改善
- [ ] タブをWAI-ARIA準拠にする。
- [ ] モーダルのキーボード操作（Esc, Tab循環, フォーカス復帰）対応。
- [ ] サムネイル導線を一本化し、情報設計を整理。
- [ ] 空状態/読み込み中/エラー状態の表示を統一。

**完了条件**
- キーボードのみで主要操作が完結。
- ユーザーテストで「目的記事に到達できる」成功率が改善。

### Phase 2（1〜2週間）: 性能・保守性改善
- [ ] `script.js` を `data`, `state`, `render`, `events` に分割。
- [ ] `innerHTML` 依存を減らし、DOM APIで安全に描画。
- [ ] API側にページング/検索パラメータを導入（可能なら）。
- [ ] SW戦略見直し（静的資産: stale-while-revalidate等）。

**完了条件**
- 記事件数増加時でも体感速度が維持。
- 変更時の影響範囲がモジュール単位で限定される。

### Phase 3（継続）: 運用最適化
- [ ] エラー監視・速度計測（Sentry/Web Vitals等）導入。
- [ ] KPI設定（検索利用率、0件率、記事クリック率、滞在時間）。
- [ ] 小さなA/BテストでUI改善を継続。

---

## 4. 具体的な最初の着手順（おすすめ）
1. まずクラッシュ要因（`#app`, `scrollSentinel`）の修正PRを最優先で切る。
2. 次に「エラー・ローディング・空状態」の3状態を統一し、ユーザー離脱を下げる。
3. その後、タブ/モーダルのアクセシビリティ対応を実施。
4. 同時並行でコード分割計画を作り、2週間以内に `script.js` 分割へ着手。

## 5. 総合評価
- **総合点: 64 / 100**
- すでに実利用可能な価値はあるが、**信頼性（落ちない設計）と保守性（変更しやすい設計）を最優先**で改善すべき段階。
- 上記Phase 0〜2を行えば、短期間で「壊れにくく、育てやすい」状態に到達可能。
