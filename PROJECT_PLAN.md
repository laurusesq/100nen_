# 百年前新聞 Webサイト 設計メモ（v1）

## 1. 目標
- 「百年前新聞」を、現代的で大手新聞社サイトに近い見た目と情報設計で提供する。
- 記事探索（検索・タグ・ジャンル・並び替え）を高速にし、回遊率を高める。
- 将来的な会員機能（コメント・ポイント・お気に入り・既読）を見据えて拡張しやすい構成にする。

---

## 2. 推奨アーキテクチャ（Cloudflare前提）

### フロントエンド
- 現行の静的構成（`index.html` + `script.js` + `style.css`）を維持しつつ、段階的にコンポーネント化。
- 将来拡張を考えると、次段階で **Next.js / Nuxt / SvelteKit** のいずれかへ移行を検討。

### バックエンド
- **Cloudflare Workers**: API層（記事取得、検索、コメント、認証チェック）。
- **Cloudflare D1**: 記事メタデータ、タグ、ジャンル、ユーザー、コメント、ポイント履歴。
- **Cloudflare R2**: 記事本文の原文JSONや画像など大きめデータ。
- **Cloudflare KV**: キャッシュ用途（人気記事、タグ一覧、検索サジェスト）。

### 認証
- 第一候補: **Cloudflare Access / Auth連携（Google, GitHub等）**
- もしくは Supabase Auth / Firebase Auth を連携。

## 3. 現状検討への回答

### 3-1. 記事データをスプレッドシートに保存してよいか
結論: **初期は可、ただし中期でDB移行を推奨**。

- スプレッドシートの利点
  - 編集が容易、非エンジニア運用しやすい。
- 欠点
  - 複合フィルタ・あいまい検索・既読管理・コメント連携で破綻しやすい。
  - データ整合性（タグ揺れ、重複）を保ちにくい。

**推奨**
1. 短期: スプレッドシートを「入稿UI」として維持。
2. 中期: 定期同期で D1 へ取り込み（正規化）。
3. 長期: 管理画面を作ってスプレッドシート依存を解消。

### 3-2. 無限スクロールはよいか
結論: **一覧には有効だが、新聞体験としてはハイブリッド推奨**。

- 長所: 回遊が増える。
- 短所: フッター到達しにくい、現在位置がわかりづらい、共有しづらい。

**推奨**
- PC: ページネーション + 「さらに読み込む」ボタン併用。
- モバイル: 無限スクロール維持（ただし 20 件ごとに区切りUI）。
- URLに `?page=3&tags=...` を持たせ、状態復元可能にする。

### 3-3. デザインが古くないか
結論: **情報密度は活かしつつ、視認性と余白を現代化する**。

改善軸:
- タイポグラフィ（本文・見出しのコントラスト強化）
- カード余白、行間、縦リズムの統一
- 色はモノトーン基調 + アクセント1色
- モバイルのタップターゲットを 44px 以上に
- ダークモード対応（任意）

## 4. 追加機能の実装方針

### 4-1. 新着記事通知
- Web Push（Service Worker） + 通知許可。
- サーバ側で「前回配信以降の新着」を抽出して配信。
- まずは「1日1回の新着まとめ通知」から開始。

### 4-2. ログイン + コメント + ポイント
- ログインユーザーのみコメント投稿可。
- コメント投稿でポイント付与（例: +5pt）。
- 将来用途を見据え、`point_events` テーブルで履歴管理。

### 4-3. ログインユーザーによるタグ編集
- 直接確定ではなく **提案制（moderation）** を推奨。
- `tag_suggestions` に保存し、管理者承認で反映。

### 4-4. あいまい検索
- 段階案:
  1. N-gram 前方一致/部分一致
  2. かな・カナ揺れ吸収
  3. typo耐性（Levenshtein距離）
- 将来的には Meilisearch / Typesense 等の検索エンジン導入も検討。

### 4-5. お気に入り
- `user_favorites(user_id, article_id, created_at)` を持つ。
- 一覧・記事詳細の双方でトグル可能にする。

### 4-6. 既読・未読
- 記事表示時に `user_reads` を更新。
- 一覧カードに既読バッジを表示。
- 未ログイン時は localStorage で簡易対応可能。

---

## 5. データモデル（最小）

- `articles(id, title, body, published_at, genre_id, source_url, created_at, updated_at)`
- `genres(id, name, sort_order)`
- `tags(id, name, wikipedia_title)`
- `article_tags(article_id, tag_id)`
- `users(id, provider, provider_user_id, display_name, created_at)`
- `comments(id, article_id, user_id, content, status, created_at)`
- `point_events(id, user_id, type, delta, ref_id, created_at)`
- `favorites(user_id, article_id, created_at)`
- `reads(user_id, article_id, read_at)`
- `tag_suggestions(id, article_id, user_id, tag_name, status, created_at)`

---

## 6. 画面/UX要件（重要）

- タグは複数選択可、ジャンルは単一選択。
- タグ名横のWikipediaアイコン押下で、該当Wikipediaページを新規タブで開く。
- キーワード検索 + 並び替え（新しい順 / 古い順）をURLクエリと同期。
- モバイルではジャンルタブを横スクロール可能にする。

---

## 7. 開発ロードマップ

### Phase 1（2〜3週間）
- UI改善（新聞風レイアウト、モバイル横スクロール）
- フィルタ/検索/並び替えのURL同期
- データ取得をAPI化（Workers）

### Phase 2（2〜4週間）
- 認証導入
- コメント、お気に入り、既読
- ポイント履歴

### Phase 3（3〜5週間）
- あいまい検索強化
- 通知機能
- タグ提案・承認ワークフロー

---

## 8. 非機能要件
- Core Web Vitals 改善（LCP, INP, CLS）
- 画像最適化（WebP/AVIF, 遅延読み込み）
- APIレスポンスのキャッシュ戦略（stale-while-revalidate）
- 監視: エラー率、検索失敗率、スクロール到達率

---

## 9. まず着手すべき具体タスク
1. 画面上の状態（タグ・ジャンル・検索・並び順）をURLクエリに反映。
2. 記事データスキーマを確定し、D1へ移行可能な形に整える。
3. タグのWikipedia遷移ロジックを関数化し、タイトル正規化を実装。
4. モバイルのジャンルバーを横スクロール + スナップ挙動にする。
5. 認証プロバイダ候補を比較して1つに絞る。
